# 权限分级安全与体验升级实施计划

---

## 第一部分：数据泄露修复（最高优先级）

审查发现 5 条泄露路径，按严重程度排序。

### P0-1：平台 Permission 单条查询/修改可越权访问应用级权限

**问题**：`GET /api/v1/permissions/{id}` 和 `PUT /api/v1/permissions/{id}` 使用 `RepositoryBase.FindByIdAsync`，仅按 `TenantId + Id` 查询，无 `AppId IS NULL` 过滤。攻击者可通过猜测/枚举 ID 查看或修改应用级权限。

**攻击链路**：

```
GET /api/v1/permissions/12345  -->  PermissionQueryService.GetDetailAsync
  -->  FindByIdAsync(tenantId, 12345)
  -->  SQL: WHERE TenantIdValue=@t AND Id=12345  (无 AppId IS NULL)
```

**修复方案**：

- 在 [IPermissionRepository.cs](src/backend/Atlas.Application/Identity/Repositories/IPermissionRepository.cs) 新增 `FindByIdPlatformOnlyAsync(TenantId, long id)` 方法
- 在 [PermissionRepository.cs](src/backend/Atlas.Infrastructure/Repositories/PermissionRepository.cs) 实现该方法，追加 `.Where(x => x.AppId == null)`
- 修改 [PermissionQueryService.cs](src/backend/Atlas.Infrastructure/Services/PermissionQueryService.cs) 的 `GetDetailAsync` 使用新方法
- 修改 [PermissionCommandService.cs](src/backend/Atlas.Infrastructure/Services/PermissionCommandService.cs) 的 `UpdateAsync` 使用新方法
- 在 [PermissionsController.cs](src/backend/Atlas.WebApi/Controllers/PermissionsController.cs) 补充 DELETE 端点时也要用平台级方法

### P0-2：平台角色可被分配应用级权限 ID（跨级提权）

**问题**：`PUT /api/v1/roles/{roleId}/permissions` 调用 `RoleCommandService.EnsurePermissionsExistAsync`，内部使用 `QueryByIdsAsync` 仅按 `TenantId + ids` 验证，不检查 `AppId IS NULL`。

**攻击链路**：

```
PUT /api/v1/roles/1/permissions  { "permissionIds": [应用权限ID] }
  -->  RoleCommandService.UpdatePermissionsAsync
  -->  EnsurePermissionsExistAsync --> QueryByIdsAsync(tenantId, ids)
  -->  校验通过（应用权限也在同表）--> 写入 RolePermission
```

**修复方案**：

- 在 [IPermissionRepository.cs](src/backend/Atlas.Application/Identity/Repositories/IPermissionRepository.cs) 新增 `QueryByIdsPlatformOnlyAsync(TenantId, ids)` 或为 `QueryByIdsAsync` 增加 `platformOnly` 参数
- 在 [PermissionRepository.cs](src/backend/Atlas.Infrastructure/Repositories/PermissionRepository.cs) 实现，追加 `.Where(x => x.AppId == null)`
- 修改 [RoleCommandService.cs](src/backend/Atlas.Infrastructure/Services/RoleCommandService.cs) 第 256-272 行 `EnsurePermissionsExistAsync` 使用平台级查询

### P0-3：Permission Code 唯一性校验未按 AppId 隔离

**问题**：`FindByCodeAsync` 仅按 `TenantId + Code` 查询。平台和应用可能有相同 Code 的权限（如 `users:view`），导致：

- 应用创建权限后，平台无法创建同 Code 权限（反之亦然）
- 平台 `PermissionCommandService.CreateAsync` 会错误地将应用权限视为已存在

**修复方案**：

- 修改 `FindByCodeAsync` 签名为 `FindByCodeAsync(TenantId, string code, long? appId = null)`
- 当 `appId == null` 时查 `AppId IS NULL`；当 `appId` 有值时查 `AppId == appId`
- 更新平台 `PermissionCommandService.CreateAsync` 传 `appId: null`
- 更新应用 `AppPermissionCommandService.CreateAsync` 传对应 `appId`

### P0-4：X-App-Id Header 注入风险（开发环境已启用）

**问题**：`appsettings.Development.json` 中 `AllowHeaderOverrideWhenAuthenticated: true`，攻击者只需携带 `X-App-Workspace: 1` + `X-App-Id: <targetAppId>`，即可跨应用访问 DynamicTable、FieldPermission、Apps 等数据。

**修复方案**：

- 确认 `appsettings.json`（生产配置）中 `AllowHeaderOverrideWhenAuthenticated` 为 `false`（若未设置需补上）
- 在 `appsettings.Development.json` 中添加注释或文档，明确标记此选项仅限受控调试场景
- 长期方案：`DynamicTablesController` 等使用 `ResolveAppId()` 的平台级 API，考虑改为显式路由参数或在 Service 层二次校验 JWT claim 中的 `app_id`

## 第二部分：角色分配面板 UX 优化

文件：[RoleAssignPanel.vue](src/frontend/Atlas.WebApp/src/components/system/roles/RoleAssignPanel.vue)

### UX-1：功能权限 Tab 增加搜索框 + 已选统计 + 描述展示

**当前问题**：

- 后端已支持 keyword 搜索（`loadPermissionOptions(keyword)`），但模板中无搜索输入框
- 权限树节点仅显示 `名称 (编码)`，无 description
- 无已选数量统计

**修复**：

- 在权限树上方（全选/清空按钮行）添加 `a-input-search`，绑定 `handlePermissionSearch`
- 在 `a-tree` 的节点渲染中使用 `#title` slot，展示权限描述（Tooltip 或副标题）
- 在全选/清空旁增加 `a-tag` 展示「已选 N 项」
- 分组名做友好映射：如 `ASSETS` -> `资产管理`（可选，通过 i18n 或硬编码映射）

### UX-2：菜单/页面分配 Tab 增加搜索框 + 辅助信息 + 已选统计

**当前问题**：

- 同权限 Tab，有搜索函数但无 UI 搜索框
- 菜单项 `path`/`icon`、页面项 `routePath`/`description` 未展示
- 无已选统计

**修复**：

- 在菜单树上方添加 `a-input-search`，绑定 `handleMenuSearch`（平台用前端过滤，应用需后端支持）
- 树节点使用 `#title` slot 展示路径信息，平台模式展示 `path`，应用模式展示 `routePath` 或 `description`
- 已选统计同上

### UX-3：数据范围 Tab 部门树增加搜索 + 展示已选部门名称

**当前问题**：

- `a-tree-select` 无 `show-search`，部门多时难以找到目标
- 自定义部门时仅显示「已选 N 个部门」，不展示部门名称

**修复**：

- `a-tree-select` 添加 `show-search` 和 `tree-node-filter-prop="title"`
- 在「当前范围预览」区域，当 `dataScope === 2` 时展示所选部门名称列表（从 `departmentTreeData` 反查）

### UX-4：字段权限 Tab 空状态引导

**当前问题**：

- 未选择动态表时显示空白表格
- 应用模式下若角色无已有字段权限配置，`dynamicTableOptions` 为空

**修复**：

- 未选择动态表时展示 `a-empty` 组件 + 引导文案「请先选择动态表以配置字段权限」
- 应用模式下加载所有可用动态表（不仅限于已有配置的），允许新增字段权限

### UX-5：静默错误改为可感知提示 + 保存后刷新主表

**当前问题**：

- `loadPermissionOptions`、`loadMenuOptions`、`loadAppPageOptions` 等 catch 为 silent
- [RolesPage.vue](src/frontend/Atlas.WebApp/src/pages/system/RolesPage.vue) 的 `handleAssignSuccess` 为空函数，保存后主表不刷新

**修复**：

- 所有 catch 中改为 `message.warning('加载数据失败，请重试')`
- `RolesPage.vue` 的 `handleAssignSuccess` 调用 `fetchData()` 刷新主表
- 与 [AppRolesPage.vue](src/frontend/Atlas.WebApp/src/pages/apps/AppRolesPage.vue)（已有 `loadRoles()`）保持一致

## 第三部分：其他模块分级评估与建议

基于全量扫描，以下模块当前缺少 AppId 但已挂在 `/apps/:appId/` 路由下，**建议优先实施分离**：


| 模块                     | 当前状态         | 建议       | 理由                                     |
| ---------------------- | ------------ | -------- | -------------------------------------- |
| FormDefinition         | 无 AppId，纯租户级 | 增加 AppId | 已挂在 `/apps/:appId/forms`，不同应用需独立表单     |
| AiWorkflowDefinition   | 无 AppId，纯租户级 | 增加 AppId | 已挂在 `/apps/:appId/workflows`，工作流应按应用隔离 |
| ApprovalFlowDefinition | 无 AppId      | 评估       | 审批流可能跨应用复用，需业务确认                       |
| DashboardDefinition    | 无 AppId      | 评估       | 仪表盘可按应用定制                              |


以下模块当前为纯平台级，**可按需延后评估**：

- **审计日志 Audit**：可增加 AppId 元数据字段，便于按应用筛选
- **告警 Alert**：可增加 AppId 支持应用级告警
- **字典 DictType**：可增加 AppId 支持应用级字典
- **通知 Notification**：可增加 AppId 支持应用级定向推送
- **参数配置 SystemConfig**：可增加 AppId 支持应用级配置
- **表格视图 TableView**：可增加 AppId 或约定 tableKey 前缀

以下模块**无需分离**，保持纯平台级：

- 租户管理、登录日志、在线会话、Webhook、插件管理、监控、定时任务、授权 License、诊断

> 注：第三部分属于架构评估，本次计划仅记录建议。是否实施需用户确认，不在本轮 todo 中。
