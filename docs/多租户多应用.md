# 多租户多应用架构（术语对齐版）

> 版本：v2.0（SEC-22）  
> 更新日期：2026-03-16  
> 术语基线：`docs/analysis/unified-terminology-glossary-v1.md`

## 一、目标与范围

本文定义平台在“多租户 + 多应用 + 应用级数据源”场景下的主对象与边界，作为产品、架构与后续实现的统一基线。

- 目标：统一 `Tenant / ApplicationCatalog / TenantApplication / TenantAppInstance / TenantDataSource / Workspace / ProjectAsset` 术语。
- 范围：租户开通、租户应用可用性、租户数据源绑定、项目模式约束、安全与验收标准。
- 非目标：不在本文直接定义数据库表结构与代码重命名策略。

## 二、核心对象与边界

### 2.1 对象定义

- **Tenant（租户）**：隔离主体，承载组织、用户、权限、配额与审计边界。
- **ApplicationCatalog（应用目录项）**：平台可供租户开通的应用定义，不代表运行实例。
- **TenantApplication（租户应用开通）**：租户与应用目录项的开通/订阅关系，承载可用状态与配置入口。
- **TenantAppInstance（租户应用实例）**：租户开通后的运行载体，承载运行参数与资源绑定。
- **TenantDataSource（租户数据源）**：租户可管理的数据连接资源，可绑定到 TenantApplication。
- **ProjectAsset（项目资产）**：Workspace 内创作资产容器，仅用于创作与资产组织，不作为数据源绑定维度。

### 2.2 关系约束（必须支持）

- 一个 Tenant 可开通多个 ApplicationCatalog（`Tenant 1..N TenantApplication`）。
- 一个 ApplicationCatalog 可被多个 Tenant 开通（`ApplicationCatalog 1..N TenantApplication`）。
- 每个 TenantApplication 必须绑定一个主 TenantDataSource（允许“继承默认模板 + 覆盖”模式）。
- 若开启项目模式，TenantAppInstance 下所有业务数据必须携带 `project_id`。
- `projectCode` 在租户应用实例范围内唯一（建议唯一键：`tenant_app_instance_id + project_code`）。

## 三、角色与权限

- 平台管理员：管理 Tenant、ApplicationCatalog、全局策略与全量租户应用开通。
- 租户管理员：管理本 Tenant 的 TenantApplication、TenantDataSource 与项目模式配置。
- 安全审计/运维：查看脱敏配置与测试记录，可执行连接测试，不可读取凭据明文。

权限拆分要求：

- “查看连接参数”与“编辑连接参数”必须拆分。
- 密码/密钥字段仅允许更新，不允许读取明文。
- 所有关键写接口必须启用 `Idempotency-Key` 与 `X-CSRF-TOKEN`。

## 四、功能需求

### 4.1 租户与应用目录项管理

- Tenant 支持新增/编辑/启用/停用。
- ApplicationCatalog 支持新增/编辑/启用/停用。
- Tenant 或 ApplicationCatalog 停用后：对应 TenantApplication 保留但不可用。

### 4.2 租户应用开通（TenantApplication）

- 支持为租户开通一个或多个应用目录项。
- 开通后生成 TenantApplication，初始状态为“待配置/待验证”。
- 推荐状态流转：`待配置 → 待验证 → 可用 → 停用/异常`。

### 4.3 租户数据源（TenantDataSource）

每个 TenantApplication 至少配置以下字段：

- 数据库类型（当前固定 SQL Server，可扩展）
- Host / Port / Database
- 认证方式（用户名密码 / 可选集成认证）
- Username / Password（仅写入）
- 安全参数（Encrypt / TrustServerCertificate / Timeout）
- 连接池参数（可选）

测试连接能力必须覆盖：

- 网络连通性
- 认证可用性
- 基础查询可执行性（如 `SELECT 1`）
- 可选最小权限校验

### 4.4 生效与变更控制

- TenantDataSource 修改后必须重新测试通过方可生效。
- 变更需记录审计差异（密码字段仅记录“已变更”）。
- 支持“保存草稿不生效”与“手动切换生效版本”（推荐）。

### 4.5 项目模式（可选）

- 项目模式仅用于业务权限与数据范围，不用于数据源绑定。
- 启用项目模式后，所有业务接口必须校验 `project_id`。
- 项目停用后禁止新增读写，历史数据保留。

## 五、接口行为约束

- 所有业务请求必须先校验 `tenant_id`，再校验 TenantApplication 可用性。
- 运行时数据源解析必须基于 `tenant_id + tenant_application_id`。
- 数据源配置可缓存，但配置变更必须立即失效或版本化刷新。
- 返回错误需兼顾可观测性与脱敏要求，不泄露连接串与凭据。

## 六、安全与合规

- 租户强隔离：Tenant A 不可访问 Tenant B 的 TenantApplication 与 TenantDataSource。
- 凭据加密存储，前端不回显，日志不输出明文。
- 连接测试、状态切换、配置变更必须全量审计。
- 建议数据库账号按最小权限创建（读写分离可选）。

## 七、验收标准

1. 一个 Tenant 可开通多个 ApplicationCatalog，同一 ApplicationCatalog 可被多个 Tenant 开通。  
2. 不同 TenantApplication 可绑定不同 TenantDataSource，互不影响。  
3. 未配置或测试失败的 TenantApplication 不可进入可用态。  
4. 凭据不回显、不出参、不入日志，且变更有审计记录。  
5. 租户管理员仅可管理本租户对象；平台管理员可管理全局对象。  
6. 启用项目模式时，业务接口必须校验 `project_id`。  

## 八、术语约束

- 禁止裸用 `Application`，必须明确为 `ApplicationCatalog / TenantApplication / TenantAppInstance`。  
- 禁止裸用 `DataSource`，统一使用 `TenantDataSource`。  
- 创作语境下使用 `ProjectAsset`，避免与组织项目域混用。
